# 场景二：批量收集 + 主人决策

> 适合：邮件量较多时，不想让 AI 自动回复每封陌生邮件，而是先汇总给你，由你决定处理方式。

## 为什么需要批处理

AI API 是有配额的。如果每封陌生邮件都立即调用 AI 回复，很快就会耗尽用量。批处理模式让你在**配额消耗**和**响应及时性**之间找到平衡：

- **信任的人**（公司同事、自己）→ 即时回复，不浪费等待时间
- **陌生人来信** → 先积累，定时汇总给你，你决定哪几封值得 AI 回复

## 工作流程

```
陌生人来信
    ↓
进入批处理队列（不立即回复）
    ↓ 每 N 小时
摘要邮件发给你
    ↓
你回复摘要邮件，用自然语言说明处理意见
    ↓
Agent 解读你的意见，逐封执行回复或跳过
    ↓
执行结果发给你确认
```

## 配置

在 `email-acl.yaml` 中启用：

```yaml
# 管理员（你自己）——始终即时处理
admin_senders:
  - me@gmail.com

# 即时回复名单——公司同事不走批处理
instant_reply_senders:
  - "@mycompany.com"
  - trusted-partner@example.com

# 批处理模式
batch_mode:
  enabled: true
  collect_interval_hours: 2    # 每 2 小时发一次摘要
  ai_profile: claude-code      # 解读你指令时用的 AI Profile
```

## 回复处理意见

直接 Reply 那封摘要邮件，用自然语言说明：

```
第2封帮我认真回一下，表示感兴趣想进一步了解。
第1封礼貌婉拒，说暂时没有合作计划。
第3封忽略。
```

或者简单一些：

```
全部礼貌婉拒
```

```
1和2回复，3忽略
```

Agent 会解读你的意图，执行完后回复一封执行结果摘要给你。

## 信任域配置建议

| 发件人类型 | 建议配置 |
|-----------|---------|
| 你自己     | `admin_senders` |
| 公司同事   | `instant_reply_senders: ["@mycompany.com"]` |
| 长期合作伙伴 | `instant_reply_senders: ["partner@example.com"]` |
| 陌生人     | 不配置（进批处理队列） |
| 已知垃圾邮件 | `denied_senders` |

::: tip 批处理 + 白名单的区别
`allowed_senders` 控制"谁能进来"（白名单之外的会被 ACL 拦截）；
`instant_reply_senders` 控制"谁立即回复"（其余进批处理队列）。
两者独立配置，语义不同。
:::
